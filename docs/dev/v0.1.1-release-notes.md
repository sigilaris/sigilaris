# Sigilaris v0.1.1 Release Notes

## Release Summary
- Release date: 2026-03-27
- Artifact version: `0.1.1`
- Scope: lock the shared transaction/codec/failure foundations introduced for the `v0.1.1` milestone.

## Highlights
- Added the canonical `Boolean` byte codec contract:
  - `false -> 0x00`
  - `true -> 0x01`
- Consolidated JSON subtype labeling for sealed traits and enums behind the shared `JsonSubtypeLabels` projection, keeping `wrapped-by-type-key` as the only public JSON sum representation.
- Introduced reusable opaque companion helpers:
  - `OpaqueValueCompanion`
  - `KeyLikeOpaqueValueCompanion`
- Migrated `GroupId` and `NetworkId` to the shared opaque helper surface and removed repeated manual codec forwarding.
- Added `TxRegistry.of[...]` with compile-time `ReducerCoverage[T]` proof checks for registered transactions.
- Added `FailureCode` / `ErrorKey` metadata while preserving existing human-readable failure message strings.
- Kept codec law support internal-only under test scope and retrofitted representative byte/json/ordered property suites to use it.

## Public Contract Notes
- `TypeNameStrategy.FullyQualified` remains a compatibility alias to the canonical Scala 3 Mirror label in `v0.1.1`; it does not introduce a second wire format.
- `TxEnvelope` and `NetworkId` remain the canonical shared transaction metadata types under `org.sigilaris.core.application.transactions`.
- Formatter APIs preserve legacy rendering:
  - `ClientFailureMessage.invalidRequest(...)`
  - `ClientFailureMessage.notFound(...)`
  - `ClientFailureMessage.forbidden(...)`
  - `ConflictMessage.format(...)`
- New metadata-aware paths are also available:
  - `ClientFailureMessage.invalidRequestWithCode(...)`
  - `ClientFailureMessage.notFoundWithCode(...)`
  - `ClientFailureMessage.forbiddenWithCode(...)`
  - `ConflictMessage.formatWithCode(...)`

## Upgrade Notes
- Update build definitions to `0.1.1`.
- If you register transaction tuples explicitly, prefer `TxRegistry.of[...]` instead of constructing registries ad hoc so missing reducer coverage fails at compile time.
- If you map failures to external protocols, start from `SigilarisFailure.code` / `ErrorKey` rather than parsing formatted message strings.
- If you define simple opaque wrappers over existing representation types, prefer the new companion helpers over repeating manual `ByteEncoder` / `ByteDecoder` / `JsonEncoder` / `JsonDecoder` forwarding.

## Verification Snapshot
- JVM targeted suites passed for:
  - codec contracts
  - opaque helper migration
  - registry coverage compile-time checks
  - failure metadata
  - accounts/group blueprint flows
  - accounts+group integration
- JS targeted suites passed for the same representative coverage, including the composed accounts+group integration flow.
