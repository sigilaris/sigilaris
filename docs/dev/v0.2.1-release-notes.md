# Sigilaris v0.2.1 Release Notes

## Release Summary
- Release date: 2026-05-16
- Artifact version: `0.2.1`
- Scope: extend the JVM HotStuff runtime baseline shipped in `v0.2.0` with an application-neutral proposal input hook, so autonomous leader proposals can carry embedder-supplied work. No other module surface changes.

## Highlights
- Added a HotStuff-owned proposal input contract that lets the autonomous pacemaker ask the embedding application for the next proposal input — block-header commitments and `ProposalTxSet` — before signing.
- The in-memory pacemaker proposal path is no longer limited to synthetic empty blocks; it can publish a non-empty `ProposalTxSet` supplied by an application provider while keeping `emittedProposalWindow` single-emission semantics.
- Legacy empty proposals remain available as an explicit `AllowLegacyEmpty` fallback policy. A `RequireProviderInput` mode is available for production embedders that want misconfiguration to fail visibly.
- Provider-supplied input is validated locally (parent / height / window / timestamp / canonical tx-set / supported tx-id / optional bounds) and then passes through the same `HotStuffValidator.validateProposal` gate used for peer proposals before any local signature is published.
- Pacemaker diagnostics expose provider outcome (`Supplied` / `NoWork` / `Rejected` / `Failed` / `Invalid`), reason / detail metadata, and a `fallbackUsed` flag without including application payload bodies. Unexpected provider exceptions record only the exception class name, never the exception message.

## Public Contract Notes
- Published coordinates are unchanged from `v0.2.0`:
  - `org.sigilaris:sigilaris-core_3`
  - `org.sigilaris:sigilaris-core_sjs1_3`
  - `org.sigilaris:sigilaris-node-common_3`
  - `org.sigilaris:sigilaris-node-common_sjs1_3`
  - `org.sigilaris:sigilaris-node-jvm_3`
- New public types in `org.sigilaris.node.jvm.runtime.consensus.hotstuff`:
  - `HotStuffProposalInputProvider[F]` — application-owned source of HotStuff proposal input.
  - `HotStuffProposalInputRequest`, `HotStuffProposalInput`, `HotStuffProposalInputBounds`.
  - `HotStuffProposalInputProviderResult` sealed taxonomy (`Supplied` / `NoWork` / `Rejected` / `Failed`).
  - `HotStuffProposalInputFallbackPolicy` (`AllowLegacyEmpty`, `RequireProviderInput`).
  - `HotStuffProposalInputRuntimeConfig` with `legacyCompatible`, `withProviderFallback`, `requireProvider`, `requireProviderInput`, and `validateForAutomaticConsensus`.
  - `LegacyEmptyHotStuffProposalInputProvider` for embedders that want to preserve the previous autonomous empty proposal explicitly.
  - `HotStuffProposalInputDiagnosticOutcome` and a new `HotStuffPacemakerDiagnostic.ProposalInputResult` case for pacemaker snapshots.
- `HotStuffNodeRuntime.create`, `HotStuffRuntimeBootstrap.fromTopology`, and `HotStuffRuntimeBootstrap.fromConfig` accept an optional `proposalInputConfig: HotStuffProposalInputRuntimeConfig[F]` parameter (default: `legacyCompatible`). The assembled bootstrap entrypoints always validate the proposal-input config against the automatic pacemaker they attach.
- No changes to HotStuff signing bytes, vote rules, QC rules, validator-set rotation semantics, or transport contracts.

## Upgrade Notes
- Update build definitions to `0.2.1`.
- Existing callers do not need to change wiring: the default `legacyCompatible` config preserves the prior autonomous empty proposal behavior.
- To drive autonomous proposals from application work, pass `HotStuffProposalInputRuntimeConfig.withProviderFallback(provider)` (keeps the legacy fallback) or `.requireProvider(provider)` (no fallback) when constructing the runtime or bootstrap.
- Consumers that pattern-match on `HotStuffPacemakerDiagnostic` should handle the new `ProposalInputResult` case; the enum is sealed, so the compiler will surface missing branches.
- Provider exception handling: provider implementations should prefer returning `HotStuffProposalInputProviderResult.Failed(reason, detail)` over throwing, since thrown exceptions are recorded with the exception class name only.
- See [HotStuff Proposal Input Provider Handoff](hotstuff-proposal-input-provider-handoff.md) for the minimal embedder adapter shape and integration smoke pattern.
